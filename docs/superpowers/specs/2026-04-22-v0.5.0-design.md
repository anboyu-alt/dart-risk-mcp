# DART 리스크 분석 MCP — v0.5.0 설계

작성일: 2026-04-22
작성자: 강반장
대상 버전: v0.5.0 (P0 릴리스)

---

## 0. 컨텍스트

본 MCP는 "**공시 기반 불공정거래 위험 모니터링**"을 목적으로 한다. v0.5.0은 2026-04 시점에 확보한 OpenDART API 개발가이드(83개 API 전체)를 참조해 기존 17개 도구·8개 위기 패턴의 탐지 완성도를 끌어올리는 단계다.

확장 범위는 **B(행위자·자금흐름·의사결정 추적에 필요한 맥락 정보) 수준**으로 고정한다. 광의의 재무·거버넌스 이상(=C)은 범위 밖.

전체 확장 로드맵(P0 → P1 → P2)을 "차례대로 검증하며 진행"하는 전략을 채택하며, 본 문서는 **P0만**을 다룬다.

## 1. 범위 & 탐지 효과

### 1.1 v0.5.0 릴리스 범위 (P0)

| 항목 | 내용 |
|---|---|
| 신규 도구 | 2개 — `track_fund_usage`, `get_major_decision` |
| 기존 도구 강화 | 3개 — `check_disclosure_risk`, `analyze_company_risk`, `build_event_timeline` |
| 사용 API 추가 | DS002 2020016/17(공모/사모자금 사용내역), DS005 2020042~2020053(12개 결정 공시) |
| 미포함 (차기) | P1: `track_capital_structure`, `scan_financial_anomaly` / P2: `get_securities_offering` |

### 1.2 금감원 패턴별 탐지 완성도 변화

현재 `taxonomy.py`의 8개 패턴 중 v0.5.0으로 커버리지가 개선되는 항목:

| 패턴 | 현재 | v0.5.0 이후 |
|---|---|---|
| `zombie_ma` (무자본 M&A) | CB 인수자만 추출 | 조달자금 **실제 사용 내역** × **타법인주식 양수결정 구조** 교차검증 |
| `fake_new_biz` (허위 신사업) | 사업목적 키워드 | 공모자금 **계획 vs 실제** 불일치 탐지 |
| `related_party_hollowing` (특수관계자 공동화) | 특수관계자 키워드 | 영업·유형자산·타법인주식 **양수도 경로 추적** |
| `delisting_evasion` (상폐 회피) | 일부 | 영업양수도·합병·분할 구조화 포착(자본구조는 P1에서 보완) |
| `audit_insider_dump` / `reverse_split_spiral` / `debt_spiral` / `founder_fade` | 유지 | 유지 |

### 1.3 릴리스 게이트

실제 금감원 적발 상장사 5건을 v0.5.0으로 재탐지했을 때, **4건 이상에서 기존 대비 신규 증거**(자금사용내역 이상 or DS005 결정 플래그)가 탐지되어야 한다.

사례 선정은 금감원 감독회보·증선위 의결서·`catalog.py` MD 카탈로그에서 **실명/접수번호가 공개된 건**만 대상으로 한다. 비공개 사례 추측은 금지.

## 2. 신규 도구 스펙

### 2.1 `track_fund_usage(company_name, lookback_years=3)`

**목적** — 조달자금의 계획 vs 실제 사용 불일치 탐지 (`zombie_ma`, `fake_new_biz` 핵심 증거)

| 항목 | 내용 |
|---|---|
| DART API | `pssrpCptalUseDtls.json`(공모 2020016), `prvsrpCptalUseDtls.json`(사모 2020017) |
| 입력 | `company_name` (str, 기업명 또는 종목코드), `lookback_years` (int, 1~5, 기본 3) |
| 내부 흐름 | `resolve_corp` → 연도 반복 호출(공모+사모) → 건별 집계 → 이상 플래그 |
| 핵심 필드 | `mtrpt_cptal_use_plan_useprps`(계획 용도), `mtrpt_cptal_use_plan_prcure_amount`(조달 금액), `real_cptal_use_dtls_cn`(실제 사용 내용), `real_cptal_use_dtls_amount`(실제 사용 금액), `dffrnc_occrrnc_resn`(차이 사유) |
| 반환 | 조달·사용 매칭 표(회차·납입일·계획용도·계획금액·실제내용·실제금액·차이사유·플래그) + 이상 건 요약 + 금감원 카탈로그 참조 |
| 신규 신호 키 | `FUND_DIVERSION`, `FUND_UNREPORTED` |

**이상 플래그 규칙**
- `FUND_DIVERSION`: 차이사유에 "목적 변경", "사업 취소", "일반 운영자금" 등 키워드 포함
- `FUND_UNREPORTED`: 계획 금액 > 0 이면서 실제 금액 == 0 (미집행) 또는 실제 사용 내용 공란

### 2.2 `get_major_decision(rcept_no, decision_type="")`

**목적** — DS005 12개 "결정" 공시를 구조화 필드로 단일 조회 (`related_party_hollowing`·`delisting_evasion` 경로 포착)

| 항목 | 내용 |
|---|---|
| DART API | 12개 엔드포인트 라우팅 — 2020042~2020053 |
| 입력 | `rcept_no` (14자리), `decision_type` (옵션) |
| 자동 판별 | 보고서명 키워드 매칭 → 엔드포인트 라우팅 ("영업양수"→2020042, "타법인 주식…양수"→2020046 등) |
| 추출 공통 필드 | 상대방(`dlptn_cmpnm`, `dlptn_rl_cmpn`), 금액, 자산대비 비율, 특수관계인 여부(`ftc_stt_atn` 등), 결의일, 외부평가 여부(`exevl_atn`) |
| 반환 | 결정 유형·상대방·금액·비율·특수관계·결의일·평가유무 + 연관 위험 신호 + `view_disclosure` 연동 안내 |
| 신규 신호 키 | `DECISION_RELATED_PARTY`, `DECISION_OVERSIZED`, `DECISION_NO_EXTVAL` |

**이상 플래그 규칙**
- `DECISION_RELATED_PARTY`: 상대방이 특수관계인 & 자산대비 10% 이상
- `DECISION_OVERSIZED`: 자산대비 30% 이상 거래
- `DECISION_NO_EXTVAL`: 외부평가 미실시 & 대규모 금액

**엔드포인트 라우팅 테이블**

| decision_type | DART API ID | 엔드포인트 |
|---|---|---|
| `business_acq` | 2020042 | 영업양수결정 |
| `business_div` | 2020043 | 영업양도결정 |
| `tangible_acq` | 2020044 | 유형자산양수결정 |
| `tangible_div` | 2020045 | 유형자산양도결정 |
| `stock_acq` | 2020046 | 타법인주식양수결정 |
| `stock_div` | 2020047 | 타법인주식양도결정 |
| `bond_acq` | 2020048 | 주권관련사채권양수결정 |
| `bond_div` | 2020049 | 주권관련사채권양도결정 |
| `merger` | 2020050 | 회사합병결정 |
| `demerger` | 2020051 | 회사분할결정 |
| `demerger_merger` | 2020052 | 회사분할합병결정 |
| `stock_exchange` | 2020053 | 주식교환이전결정 |

## 3. 기존 도구 강화

원칙: 기존 호출 시그니처·반환 구조는 하위호환 유지, 신규 API는 "추가 섹션" 또는 "신호 풍부화"로 덧붙임.

### 3.1 `check_disclosure_risk(rcept_no, report_name)`

- 보고서명이 DS005 12종 "결정" 중 하나로 판별되면 내부에서 `get_major_decision` 호출 → 상대방·금액·특수관계 구조화 결과를 응답에 포함
- 기존 CB/BW 인수자 추출은 유지 (분기 처리: CB/BW → 기존, 양수도·합병·분할 → 신규)
- 탐지 신호에 `DECISION_*` 3개 키가 자동 반영

### 3.2 `analyze_company_risk(company_name, lookback_days=90)`

- 탐지된 공시 중 DS005 결정 공시가 있으면 `get_major_decision`으로 구조화 필드 자동 추출 → "주요 결정 상대방" 섹션 추가
- `track_fund_usage` 결과 1회 호출해 리포트 말미에 요약 표시 (자금사용내역은 정기보고서 연 단위 갱신이므로 `lookback_years=3` 고정)
- 신규 신호가 `zombie_ma`·`fake_new_biz`·`related_party_hollowing` 패턴 매칭 정확도 상승에 기여
- **호출 상한**: DS005 결정 공시 최신 10건으로 제한 (초과 시 안내 메시지)

### 3.3 `build_event_timeline(company_name, lookback_days=365)`

- 신호 → 단계 매핑 확장:
  - 진입기에 `DECISION_RELATED_PARTY`, `FUND_DIVERSION` 추가
  - 심화기에 `DECISION_OVERSIZED`, `DECISION_NO_EXTVAL` 추가
- 이벤트 표시에 "상대방" 행 추가 (기존 "CB 인수자"와 동일 형식)
- `CROSS_SIGNAL_PATTERNS`는 수정하지 않음 (기존 8개 유지, 신규 키가 기존 패턴에 편입됨)

### 3.4 공통 변경

| 파일 | 변경 |
|---|---|
| `signals.py` | 신규 5개 신호(`FUND_DIVERSION`, `FUND_UNREPORTED`, `DECISION_RELATED_PARTY`, `DECISION_OVERSIZED`, `DECISION_NO_EXTVAL`) 추가 + `SIGNAL_KEY_TO_TAXONOMY` 매핑 |
| `taxonomy.py` | 해당 신호가 들어갈 taxonomy ID의 `indicators` 보강. 패턴 수정 없음 |
| `core/__init__.py` | 신규 함수 export |
| `server.py` | `@mcp.tool()` 2개 등록, 기존 3개 도구 함수 내부만 수정 |

## 4. 데이터 흐름 · 캐시 · 에러 처리

### 4.1 데이터 흐름

```
[track_fund_usage]
 company_name → resolve_corp(cache 24h) → corp_code
  → 연도 반복 (N = lookback_years):
     ├─ GET /api/pssrpCptalUseDtls.json (공모)
     └─ GET /api/prvsrpCptalUseDtls.json (사모)
  → 건별 정규화 → 이상 플래그 판정 → 응답 포맷

[get_major_decision]
 rcept_no → (decision_type 자동 판별: list.json 캐시 or 보고서명 매칭)
  → 라우팅 테이블에서 엔드포인트 선택
  → GET /api/{endpoint}.json
  → 구조화 필드 추출 → 플래그 판정 → 응답 포맷

[analyze_company_risk 강화]
 기존 흐름 그대로 + 리포트 조립 단계에서:
  ├─ 최신 DS005 결정 공시 ≤10건 → get_major_decision 호출
  └─ track_fund_usage 1회 호출
```

### 4.2 캐시

| 대상 | 위치 | TTL | 정책 |
|---|---|---|---|
| corp_code | 기존 파일 캐시 | 24h | 재사용, 변경 없음 |
| 공시 ZIP | 기존 인메모리 LRU | 10분 | 재사용, 변경 없음 |
| `track_fund_usage` 결과 | 신규 인메모리 LRU | 10분, 최대 20건 | 키 = `(corp_code, lookback_years)` |
| `get_major_decision` 결과 | 신규 인메모리 LRU | 10분, 최대 50건 | 키 = `rcept_no` |

캐시는 `dart_client.py` 모듈 전역 OrderedDict (외부 라이브러리 없이 기존 `_zip_cache` 패턴 복제).

### 4.3 에러 처리

| 상황 | 처리 |
|---|---|
| `status` ≠ "000" | 해당 건 빈 결과로 치환, 상위 집계에 영향 X |
| `013` 데이터 없음 | 정상 케이스 — 빈 리스트 반환, 경고 없음 |
| `020` 요청 제한 초과 | `_retry` 지수 백오프(기존), 3회 실패 시 부분 결과 + 경고 |
| 네트워크 예외 | `requests.RequestException` 잡아 빈 결과, 도구 레벨 전파 금지 |
| `decision_type` 자동 판별 실패 | "보고서명으로 유형을 판별할 수 없습니다. `decision_type`을 명시해 주세요." + 지원 타입 목록 |
| `analyze_company_risk`에서 부분 실패 | 성공 건만 리포트, 실패 건은 "N건 조회 실패" 한 줄 고지 |

### 4.4 호출수 가드

| 도구 | 최대 외부 호출 수 |
|---|---|
| `track_fund_usage` | `lookback_years × 2` (최대 10) |
| `get_major_decision` | 1 |
| `analyze_company_risk` 강화 후 | 기존 + 최대 10(DS005) + 최대 6(fund_usage) |

## 5. 테스트 · 검증 전략

### 5.1 단위 테스트 (requests 모킹)

**`track_fund_usage`**

| 케이스 | 확인 포인트 |
|---|---|
| 계획 100억 / 실제 100억 | 플래그 없음 |
| 계획 > 0, 실제 == 0 | `FUND_UNREPORTED` 플래그 |
| `dffrnc_occrrnc_resn` = "일반 운영자금으로 변경" | `FUND_DIVERSION` 플래그 |
| `status=013` | 빈 리스트, 에러 X |
| 네트워크 실패 | 빈 리스트, 예외 전파 없음 |
| 연도 3회 반복 중 1회만 성공 | 성공분만 집계 |

**`get_major_decision`**

| 케이스 | 확인 포인트 |
|---|---|
| 보고서명 "타법인 주식…양수결정" | 2020046 라우팅 |
| 특수관계인 & 자산대비 15% | `DECISION_RELATED_PARTY` + `DECISION_OVERSIZED` 동시 |
| 외부평가 미실시 & 대규모 금액 | `DECISION_NO_EXTVAL` |
| `decision_type` 판별 실패 | 안내 메시지 + 지원 타입 목록 |
| 잘못된 `rcept_no` | 입력 검증 에러 |

**공통**
- 캐시 hit/miss: 같은 키로 2회 호출 시 외부 API 1회만
- 호출수 상한: `analyze_company_risk`에서 DS005 15건 있을 때 정확히 10건만

### 5.2 통합 검증 (실제 DART API)

**회귀 시나리오** — 삼성전자로 강화 전/후 `analyze_company_risk` 실행, 기존 신호 출력 동일함을 확인.

**검증 시나리오** — 금감원 적발 상장사 5건 재탐지:

| 사례 유형 | 선정 기준 | 기대 결과 |
|---|---|---|
| `zombie_ma` 사례 2건 | 무자본 M&A 증선위 의결 공개 | `FUND_DIVERSION` or `DECISION_RELATED_PARTY` 중 1개 신규 탐지 |
| `fake_new_biz` 사례 1건 | 허위 신사업 과징금 부과 | `FUND_DIVERSION` 탐지 |
| `related_party_hollowing` 사례 1건 | 특수관계자 부당지원 공시 | `DECISION_RELATED_PARTY` 탐지 |
| 추가 1건 | 카탈로그 등록 사례 | 기존 대비 신규 증거 1개 이상 |

**릴리스 게이트**: 5건 중 4건 이상에서 기존 대비 신규 증거 탐지 → v0.5.0 승인.

### 5.4 미커버 영역 (명시적 한계)

- **API 응답 필드 버전 차이**: 2018-01-18 이전 공시는 `cptal_use_plan`(구필드), 이후는 `mtrpt_cptal_use_plan_*`(신필드). 둘 다 파싱하되, 이상 판정은 신필드 공시에만 적용.
- **특수관계인 판정**: DART 필드에 명시가 없으면 false 간주(과소탐지 감수). 이름 기반 추정은 하지 않음.
- **`track_fund_usage`는 정기보고서 주요정보**: 사업/반기/분기보고서 제출 시점에만 갱신. 실시간 자금 이동 추적은 범위 밖.
