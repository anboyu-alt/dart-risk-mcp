# v0.6.0 — 자본구조 추적 + 재무 이상 스캔

**설계일**: 2026-04-22
**대상 릴리스**: v0.6.0 (v0.5.0 이후 첫 마이너)
**연관 스펙**: `2026-04-22-v0.5.0-design.md` (v0.5.0 완료됨)

---

## 1. 배경과 목표

v0.5.0은 **공시된 개별 이벤트**(자금사용·주요결정)에 이상 플래그를 붙였다. 그러나 금감원 적발 사례를 다시 돌려본 결과(V.2 검증) 두 가지 공백이 드러났다:

1. **이벤트 간 리듬 부재** — 12개월에 3자배정 3건, 감자 1건, 자사주 2건이 연달아 발생하는 "자본 주무르기" 패턴 자체가 리포트에 집계되지 않음. 개별 공시는 모두 탐지되지만 "리듬"이 신호로 올라오지 않음.
2. **공시 이전 조짐 부재** — 매출채권·재고자산 급등, 영업현금흐름·순이익 괴리 같은 재무 이상은 **공시가 나기 몇 분기 전부터** 숫자로 드러난다. 현재 도구는 공시 중심이라 이 선행 지표를 놓침.

v0.6.0 목표:
- 자본 이벤트의 **시간축 집계**로 "주무르기 리듬" 탐지
- 재무제표 4개 핵심 지표의 **YoY 비교**로 분식 초기 조짐 탐지
- 두 결과를 `analyze_company_risk`에 자동 통합해 사용자 추가 호출 없이도 증거 확보

## 2. 스코프와 설계 원칙

### 2.1 범위

| 포함 | 제외 |
|---|---|
| 자본 이벤트 시간축 집계 (`track_capital_structure`) | 자본 이벤트의 정확한 금액·희석률 계산 (→ 기존 `get_major_decision` 위임) |
| 재무제표 4개 지표 YoY 이상 탐지 (`scan_financial_anomaly`) | XBRL 주석 파싱, 업종별 임계값 보정, 회계감사보고서 분석 |
| `analyze_company_risk` 자동 통합 | `check_disclosure_risk` 변경 (개별 공시 단위라 무관) |
| 9개 자본 이벤트 타입 (증자·감자·자사주·CB·BW·EB·RCPS 등) | 무상증자 (희석 아님, YAGNI) |

### 2.2 설계 원칙

- **YAGNI**: 정확한 희석률 소수점은 계산하지 않는다. "몇 번 주물렀나"의 리듬이 신호다.
- **v0.5.0 정합성**: 신규 DART 엔드포인트 0개, 신규 외부 의존성 0개. 기존 `fetch_company_disclosures`·`fetch_financial_statements`만 사용.
- **회귀 격리**: 기존 18개 테스트·19개 도구 시그니처 변경 없음. 재무 API 실패 시 해당 섹션만 스킵.
- **임계값은 초기값에서 시작**: 실데이터로 감을 잡은 뒤 다음 마이너에서 조정.

## 3. 구현 요소

### 3.1 신규 MCP 도구 2개 (총 19개 → 21개)

#### `track_capital_structure(company_name: str, lookback_years: int = 3) -> str`

자본 이벤트(증자·감자·자사주·CB/BW/EB/RCPS 등)를 시간순으로 집계해 "자본 주무르기" 리듬을 탐지.

- `lookback_years` 범위: 1~5 (밖이면 3으로 강제)
- 내부 흐름: `resolve_corp` → `fetch_company_disclosures(lookback_years * 365)` → `match_signals` × N → `CAPITAL_EVENT_KEYS` 필터 → 정정공시 제외 → `detect_capital_churn` → 포맷
- 반환: 총 이벤트 수, 12개월 최대 집중도, 연도별 집계, 시계열 목록, `CAPITAL_CHURN` 플래그 여부

#### `scan_financial_anomaly(company_name: str, year: str = "", report_type: str = "annual") -> str`

재무제표 4개 지표를 YoY로 비교해 분식·부실 초기 조짐을 탐지.

- `year` 빈 값이면 직전 연도. `report_type`: `annual` | `half` | `q1` | `q3`
- 내부 흐름: `resolve_corp` → `fetch_financial_statements` 1회 호출 (응답에 당기·전기 포함) → `detect_financial_anomaly` → 포맷
- 반환: 지표별 당기·전기·Δ·판정 표 + 플래그 목록

### 3.2 신규 내부 함수 2개 (`dart_client.py`)

```python
def detect_capital_churn(
    events: list[dict],
    lookback_years: int,
) -> dict:
    """
    12개월 슬라이딩 윈도우에서 자본 이벤트 ≥3건이면 CAPITAL_CHURN 플래그.

    events: match_signals가 반환한 신호 이벤트 리스트 (key, rcept_dt 등 포함)
    반환: {
        "max_12m_count": int,      # 12개월 윈도우 최대 집중도
        "total_events": int,       # 자본 이벤트 전체 수
        "by_year": dict[str, int], # {"2024": 3, "2025": 4}
        "events": list[dict],      # 필터된 자본 이벤트 (시간 오름차순)
        "flags": list[str],        # ["CAPITAL_CHURN"] or []
    }
    """


def detect_financial_anomaly(
    current: dict[str, int],
    prior: dict[str, int],
) -> tuple[list[str], list[dict]]:
    """
    당기·전기 재무 dict를 받아 4개 지표 판정.

    current/prior: {"매출액": int, "매출채권": int, "재고자산": int,
                    "영업활동현금흐름": int, "당기순이익": int,
                    "자본총계": int, "자본금": int}
    반환: (flags, metrics)
        flags: ["AR_SURGE", "CASH_GAP", ...] 0~4개
        metrics: [{"name": "매출채권/매출", "current": 78.0, "prior": 22.0,
                   "delta": 56.0, "unit": "%", "flagged": True}, ...]
    """
```

### 3.3 신규 신호 키 5종 (`signals.py`·`taxonomy.py`)

| 키 | 라벨 | taxonomy ID | 트리거 |
|---|---|---|---|
| `CAPITAL_CHURN` | 자본 이벤트 과다 반복 | 2.7 (신규) | 12개월 내 자본 이벤트 ≥3건 |
| `AR_SURGE` | 매출채권/매출 비율 급등 | 6.1 (기존) | YoY Δ ≥ +50%p |
| `INVENTORY_SURGE` | 재고자산/매출 비율 급등 | 6.1 (기존) | YoY Δ ≥ +50%p |
| `CASH_GAP` | 순이익·현금흐름 괴리 | 6.1 (기존) | 순이익 > 0 AND 영업현금흐름 < 0 |
| `CAPITAL_IMPAIRMENT` | 자본잠식 근접 | 8.2 (기존) | 자본총계/자본금 < 50% |


### 3.5 재무 계정과목 매핑 (하드코딩 fallback)

DART `fnlttSinglAcnt.json` 응답 필드의 `account_nm`을 순서대로 탐색:

| 논리명 | fallback 우선순위 |
|---|---|
| 매출 | `매출액` → `영업수익` |
| 매출채권 | `매출채권` → `매출채권및기타채권` |
| 재고자산 | `재고자산` |
| 영업현금흐름 | `영업활동현금흐름` → `영업활동으로인한현금흐름` |
| 당기순이익 | `당기순이익` → `당기순이익(손실)` |
| 자본총계 | `자본총계` |
| 자본금 | `자본금` |

못 찾은 필드 관련 지표는 `"데이터 없음"`으로 스킵, 에러 반환 안 함.

## 4. 아키텍처

### 4.1 호출 흐름

```
[track_capital_structure 단독]
 company_name
  → resolve_corp
  → fetch_company_disclosures(lookback_years × 365)
  → match_signals × N
  → CAPITAL_EVENT_KEYS 필터 + 정정공시 제외
  → detect_capital_churn
  → 포맷 출력

[scan_financial_anomaly 단독]
 company_name
  → resolve_corp
  → fetch_financial_statements(year, report_type)
  → 당기·전기 재무 dict 추출 (fallback 계정명 적용)
  → detect_financial_anomaly
  → 포맷 출력

[analyze_company_risk 강화]
 기존 v0.5.0 흐름 유지 +
  ├─ signal_events 수집 완료 후, calculate_risk_score 호출 전:
  │   ├─ detect_capital_churn(signal_events, lookback=1) 호출
  │   │   └─ CAPITAL_CHURN 플래그 있으면 signal_events에 합산
  │   └─ fetch_financial_statements + detect_financial_anomaly 호출
  │       └─ 4개 플래그 각각 signal_events에 합산
  ├─ 리포트 조립:
  │   ... (기존 섹션) ...
  │   [v0.6.0 신규] 자본 변동 타임라인 (최근 12개월만, 요약 버전)
  │   [v0.6.0 신규] 재무 이상 스캔 (플래그된 지표만)

[build_event_timeline 확장]
 _PHASE_MAP에 신규 키 매핑 추가:
   "CAPITAL_CHURN" → "심화기"
   "AR_SURGE" → "심화기"
   "INVENTORY_SURGE" → "심화기"
   "CASH_GAP" → "탈출기"
   "CAPITAL_IMPAIRMENT" → "탈출기"
 재무 징후(AR_SURGE 등)는 공시 이벤트가 아닌 스칼라라 event tuple 끝이 아닌 "재무 징후" 별도 블록으로 타임라인 말미에 렌더링.
```

### 4.2 캐시

신규 캐시 없음. 모든 데이터는 기존 캐시(공시 목록·재무제표)에 위임. `fetch_financial_statements`는 호출당 응답이 2~5KB로 작아 별도 캐시 불필요.

### 4.3 오류 처리

| 상황 | 동작 |
|---|---|
| `resolve_corp` 실패 | 에러 메시지 반환, 종료 |
| `fetch_company_disclosures` 빈 결과 | `track_capital_structure`: 빈 이벤트 표시, 플래그 없음 |
| `fetch_financial_statements` status 013/020 | `scan_financial_anomaly`: "재무제표 조회 불가" 표시 |
| 재무 필드 일부 누락 | 해당 지표만 "데이터 없음", 나머지 정상 계산 |
| `analyze_company_risk` 내부 재무 호출 실패 | 재무 섹션만 스킵, 나머지 리포트 정상 출력 |

## 7. 검토한 대안

### 방식 B: 신호 집계 + DS엔드포인트 상세 조회 (기각)
- 각 자본 이벤트마다 해당 DS 엔드포인트를 추가 호출해 발행가·주식수·희석률 구조화 추출
- 장점: 정확한 희석률 계산
- 기각 이유: 새 엔드포인트 래퍼 10+ 개 필요, 캐시 3개 추가, 복잡도 v0.5.0의 2배. 사용자 실제 요구는 "주무르기 리듬" 탐지이지 희석률 소수점이 아님. 필요해지면 v0.7.0에서 선택적 추가 가능.

### 방식 C: 풀 구조화 분석 (기각)
- XBRL 주석까지 파싱해 20+ 지표 전부 분석
- 기각 이유: 별도 대형 프로젝트급, 단일 마이너 릴리스 범위 초과.

### V.4 관찰 사항·후속

- **재무 이상 플래그 0건의 가능성**:
  1. 5개 표본 중 바이오(헬릭스미스·네오이뮨텍·셀리버리)는 매출 규모 자체가 작아 "전기 대비 50%p 급증" 기준을 채우기 어렵다.
  2. `fetch_financial_statements`는 감사받은 연간/반기 재무만 조회하므로, 정정·재작성 이전 원장 값이 들어와 구조 이상이 평탄화될 수 있다.
  3. 동성제약처럼 이미 자본잠식 국면에 진입한 기업도 직전년 대비 악화폭이 줄어들면(바닥 고착) 플래그가 걸리지 않는다.
- **후속 관찰 제안** (본 릴리스에서는 **코드 변경 없음**):
  - AR_SURGE / INVENTORY_SURGE 임계값 50%p → 40%p 완화 검토 (P2).
  - CASH_GAP 정의를 "영업CF < 0 + 순이익 > 0" 외에 "영업CF 대비 당기순이익 괴리율 ≥ x%" 보조 기준과 병행하는 방안.
  - CAPITAL_CHURN 은 대형주 루틴 자사주와 부실기업 연속 증자를 분리하기 위한 **규모/업종 필터** 필요.
